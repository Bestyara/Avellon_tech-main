  - `BoreHoleDialog.accept_action()` в [`/Users/anikitin/PycharmProjects/Avellon_tech-main_latest/main_window.py`](/Users/anikitin/PycharmProjects/Avellon_tech-main_latest/main_window.py) в конце вызывает `self.borehole.save_info_to_file()`.

## Цели миграции (по договорённостям)
- **Данные файлов**: парсим CSV и храним в БД (как JSON в `files.data`).
- **Миграция существующих проектов**: не делаем (на старые папки не рассчитываем), начинаем с новых проектов.

## Ключевая архитектурная идея
Сделать БД «источником истины», но переходить постепенно через слой доступа к данным:
- Ввести тонкий слой `repository/service` поверх `DbStorage`, чтобы UI/логика не работали напрямую с SQL.
- На переходный период оставить filesystem только как:
  - место для экспорта картинок/отчётов (`save_data`)
  - (временно) как “кэш/темп” при необходимости для графиков, если часть кода всё ещё ожидает путь к CSV.

## Этап 1 — Создание/открытие проекта через БД (без `info.txt`)
- **Расширить схему БД** (минимально, чтобы проект можно было открыть без `info.txt`):
  - `projects`: добавить `project_path` (TEXT, UNIQUE) и, опционально, `created_at`.
  - привести типы id к совместимым с текущими `uuid4()` (рекомендуемо: `TEXT PRIMARY KEY`), либо добавить surrogate key + хранить `uuid` отдельным полем.
  - Поскольку “проект=скважина”, при создании проекта создавать и запись в `boreholes`.
- **Добавить методы в `DbStorage`**:
  - `create_project(name, path) -> project_id`
  - `get_project_by_path(path) -> project`
  - `get_or_create_borehole_for_project(project_id, borehole_name, ...)`
- **Изменить UI создания проекта**:
  - В `CreateProjectDialog.accept_action()` после проверки имени/папки — создать запись проекта в БД.
  - Опционально: продолжать создавать папку проекта (чтобы не ломать привычный UX), но **не создавать `info.txt`**.
- **Изменить логику открытия проекта**:
  - Сейчас `Открыть проект` ожидает папку с `info.txt` (см. README и проверки в UI). Нужно заменить на:
    - либо выбор проекта из списка БД
    - либо выбор папки и проверку “папка зарегистрирована в БД”
  - Для “новые проекты только” предпочтительно: **выбор папки → если нет записи в БД, показываем предупреждение**.
- **Кэш последнего проекта**:
  - сейчас кэшируется путь через `__avellon_cache__/info.txt` (`CACHE_FILE_INFO_PATH`). На этапе 1 можно оставить путь, но правильнее начать кэшировать `project_id`.

## Тест-план (минимальный)
- Создать новый проект → закрыть приложение → открыть снова → проект открывается из БД.
- Добавить секции/шаги → перезапуск → структура восстановилась из БД.
- Добавить CSV → построить осциллограмму/таблицу максимумов → данные берутся из БД.

## Риски и оговорки
- Хранение больших массивов в JSON внутри SQLite может раздувать `db_storage.dat`. На старте ок, но дальше может понадобиться:
  - компрессия JSON
  - или нормализация в отдельную таблицу с точками/чанками
- Текущие id в коде — `uuid4()`, а в схеме БД — `bigint`. Потребуется согласование типов на самом раннем этапе.

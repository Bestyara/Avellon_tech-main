### Цель

**Сделать так**, чтобы после первого чтения CSV-файла его содержимое сохранялось в таблице `files` SQLite, а все последующие чтения данных для графиков происходили из БД (если данные там уже есть), без изменения поведения UI и логики графиков.

### Ключевые идеи

- **Схема БД**: использовать уже существующую таблицу `files` в `[db_storage.py](db_storage.py)`, расширив её полем с путём к файлу и, при необходимости, подняв версию схемы.
- **Точка интеграции**: не трогать большую часть бизнес-логики в `[borehole_logic.py](borehole_logic.py)` и окнах, а встроить работу с БД непосредственно в класс `XYDataFrame` в `[graph_widget.py](graph_widget.py)`.
- **Поведение**:
  - При первом чтении файла с диска — парсим CSV, как сейчас, и дополнительно сохраняем результат (заголовок и массив `y`) в таблицу `files`.
  - При повторных чтениях — сначала пытаемся взять данные из `files` по пути файла; если нашли и успешно распарсили JSON, больше не читаем CSV с диска.
  - Если БД по какой-то причине недоступна или данных нет — плавно откатываемся к старому поведению (чтение с диска).

### Изменения в схеме БД

1. **Обновление `db_storage.py`**
  - В методе `_create_schema_v1` (или новой версии схемы) для таблицы `files` добавить поле `file_path TEXT UNIQUE`:
    - Используется как ключ для связи между физическим CSV-файлом и записью в БД.
    - Оставить существующие поля (`file_id`, `file_name`, `borehole_id`, `part_of_file_id`, `creation_date`, `data JSON NOT NULL`).
2. **Сервисные методы для файлов**
  - Добавить методы работы с таблицей `files`:
    - `get_file_by_path(file_path: str) -> Optional[Dict[str, Any]]` — поиск записи по нормализованному пути.
    - `upsert_file_data(borehole_id: str, file_path: str, file_name: str, part_of_file_id: int, payload: Dict[str, Any]) -> str`:
      - Если запись для `file_path` существует — обновить `data` и метаданные.
      - Если нет — создать новую запись с новым `file_id`.
  - Нормализовать путь через `pathlib.Path(...).resolve()` по аналогии с `_normalize_project_path`, чтобы ключ был стабильным.

### Логика хранения содержимого файла

1. **Расширение `XYDataFrame` в `graph_widget.py`**
  - Добавить в `XYDataFrame` ленивый доступ к БД:
    - Статический метод `_get_db()`:
      - Пытается создать/вернуть единственный экземпляр `DbStorage` (по умолчанию использует `db_storage.dat`).
      - При ошибке инициализации возвращает `None`, чтобы не ломать приложение.
  - В `__init__` добавить этап чтения из БД **до** проверки `os.path.exists`:
    - Нормализовать путь файла.
    - Обратиться к `DbStorage.get_file_by_path`.
    - Если нашли запись и смогли распарсить `data` (JSON):
      - Восстановить `self.header`, `self.data`, `self.max_y`, `self.min_y`, `self.mean_y` из сохранённой структуры.
      - Установить `self.active = True` и пропустить чтение CSV.
  - Если записи в БД нет или JSON повреждён — выполнить существующую логику:
    - Проверка наличия файла.
    - `pd.read_csv` + `header_init` + `data_init`.
  - После успешного чтения CSV добавить сохранение в БД:
    - Определить `project_path` по директории файла (родитель каталога скважины):
      - Структура пути: `<...>/projects/<project_name>/<section>/<step>/<file>`.
      - `project_path` = `<...>/projects/<project_name>` = директория на один уровень выше секций.
    - Через `DbStorage.get_project_by_path(project_path)` получить `project_id`.
    - Через `DbStorage.get_or_create_borehole_for_project(project_id, borehole_name)` (где `borehole_name` = `os.path.basename(project_path)`) получить `borehole_id`.
    - Сформировать `payload` для поля `data`:
      - `{"header": ..., "data": self.data, "stats": {"max_y": ..., "min_y": ..., "mean_y": ...}}`.
      - При необходимости добавить `measurement_num` и `sensor_num`, рассчитанные из имени файла (через уже существующую функцию `get_num_file_by_default` или аналог в `borehole_logic.py`).
    - Вызвать `upsert_file_data(...)`, передавая нормализованный `file_path` и `file_name`.
2. **Формат JSON в столбце `data`**
  - Определить и зафиксировать простой формат:
    - `header`: словарь в том виде, как его возвращает текущий `header_init`.
    - `data`: словарь `{"y": [...]} после сдвига относительно среднего, как сейчас в` data_init`.
    - `stats`: `{"max_y": float, "min_y": float, "mean_y": float}`.
    - `meta`: `{"measurement_num": int, "sensor_num": int}` (опционально, для будущих задач).
  - При загрузке из БД `XYDataFrame` должен уметь работать даже если каких-то полей нет (например, старые записи) — с безопасными `get()` и дефолтами.

### Сохранение существующего поведения

1. **Совместимость и отказоустойчивость**

- Если:
  - БД недоступна,
  - нет записи для `file_path`,
  - или JSON в `data` не соответствует ожидаемому формату,
  - то `XYDataFrame` **тихо откатывается** к старой логике чтения CSV с диска.
- Вся остальная логика (`Borehole`, `Section`, `Step`, графики в `[borehole_logic.py](borehole_logic.py)` и виджеты в `[main_window.py](main_window.py)`) остаётся без изменений.
- Графики и UI продолжают работать так же, как сейчас; единственное отличие — второе и последующие чтения тех же файлов будут идти из БД.
